## Summary

All three tools are confirmed available and their JSON output schemas are fully verified by running them against the sentrux codebase. The core integration pattern from Phase 2 (`run_pmat_command<T>` in `pmat_adapter.rs`) extends cleanly for `pmat analyze graph-metrics`. Cargo-llvm-cov and clippy use `std::process::Command` directly since they are not PMAT subcommands.

**Critical schema finding — graph-metrics path format:** `pmat analyze graph-metrics --format json` returns node `name` as a bare filename (e.g., `"mod.rs"`, `"channels.rs"`) — NOT a full or relative path. This means the graph-metrics output cannot be joined to the snapshot's file-path index without disambiguation for files with the same name. The workaround is to use the snapshot's own import graph (which uses full relative paths) to build a PageRank/centrality map — or to store the graph-metrics data indexed by filename-only and display it on a best-effort basis noting the limitation. The detail panel approach (click a file, look up by filename) is workable because the user selects a specific file, so the filename is known.

**Coverage format decision:** Use `--json --summary-only --ignore-run-fail`. The `--summary-only` flag skips per-line hit data (which would be massive) and gives per-file `lines.percent` and `functions.percent` in a compact format. The `--ignore-run-fail` flag is required because sentrux currently has 27 failing tests — without it, cargo-llvm-cov exits before writing output.

**Clippy run timing decision:** Run clippy automatically during scan (it takes ~5 seconds on a warm build, already cached in this repo). It produces per-file warnings as a `--message-format=json` stream. Parse inline without a temp file.

**Risk ColorMode formula:** `risk = pagerank_rank * (1 - coverage_pct) * (1 + log(clippy_count + 1))` normalized 0.0–1.0. Fallback gracefully: if coverage unavailable, use `pagerank_rank * (1 + log(clippy_count + 1))`.

**Primary recommendation:** Three separate adapters in `pmat_adapter.rs` — `run_graph_metrics`, `run_coverage`, `run_clippy_warnings` — all following the existing `run_pmat_command<T>` subprocess pattern. New report types in `pmat_types.rs` for each. `ScanReports` gains three new `Option<T>` fields. Coverage runs on-demand via a new toolbar button.

### Pattern 2: cargo-llvm-cov Subprocess (stdout, not temp file)

cargo-llvm-cov with `--json` writes to stdout (or `--output-path` for a file). Use `--output-path` for the same temp-file consistency as PMAT:

```rust
// sentrux-core/src/analysis/pmat_adapter.rs
pub fn check_llvm_cov_available() -> bool {
    Command::new("cargo")
        .args(["llvm-cov", "--version"])
        .stdout(Stdio::null())
        .stderr(Stdio::null())
        .status()
        .map(|s| s.success())
        .unwrap_or(false)
}

pub fn run_coverage(root: &str, scan_gen: u64) -> Option<CoverageOutput> {
    let tmp_path = std::env::temp_dir().join(
        format!("sentrux_coverage_{}_{}.json", std::process::id(), scan_gen)
    );

    // --ignore-run-fail: write output even if tests fail (common in dev)
    // --summary-only: skip per-line data (too large), just per-file summary
    let _status = Command::new("cargo")
        .args([
            "llvm-cov",
            "--json",
            "--summary-only",
            "--ignore-run-fail",
            "--output-path",
            tmp_path.to_str()?,
        ])
        .current_dir(root)
        .stdout(Stdio::null())
        .stderr(Stdio::null())
        .status()
        .ok()?;

    let json_bytes = std::fs::read(&tmp_path).ok()?;
    let _ = std::fs::remove_file(&tmp_path);
    serde_json::from_slice(&json_bytes).ok()
}
```

### Pattern 3: cargo clippy --message-format=json (streaming, no temp file)

Clippy produces NDJSON lines to stdout. Parse line by line:

```rust
// sentrux-core/src/analysis/pmat_adapter.rs
pub fn run_clippy_warnings(root: &str) -> Option<ClippyReport> {
    let output = Command::new("cargo")
        .args([
            "clippy",
            "--message-format=json",
            "--",
            "-W", "clippy::pedantic",
            "-W", "clippy::nursery",
        ])
        .current_dir(root)
        .stdout(Stdio::piped())
        .stderr(Stdio::null())
        .output()
        .ok()?;

    let mut by_file: HashMap<String, FileClippyData> = HashMap::new();
    for line in output.stdout.split(|&b| b == b'\n') {
        if line.is_empty() { continue; }
        let obj: serde_json::Value = serde_json::from_slice(line).ok()?;
        if obj["reason"] != "compiler-message" { continue; }
        let msg = &obj["message"];
        if msg["level"] != "warning" { continue; }
        let lint_id = msg["code"]["code"].as_str().unwrap_or("").to_string();
        if lint_id.is_empty() { continue; }
        for span in msg["spans"].as_array().unwrap_or(&vec![]) {
            if span["is_primary"].as_bool() != Some(true) { continue; }
            let fname = span["file_name"].as_str().unwrap_or("").to_string();
            let entry = by_file.entry(fname).or_default();
            entry.total += 1;
            *entry.by_category.entry(lint_category(&lint_id)).or_insert(0) += 1;
        }
    }
    Some(ClippyReport { by_file })
}
```

### Pattern 4: Risk ColorMode Formula

Combines three signals into a single 0.0–1.0 risk score:

```rust
// sentrux-core/src/renderer/colors.rs
pub fn risk_color(
    pagerank: Option<f64>,
    coverage_pct: Option<f64>,
    clippy_count: Option<u32>,
) -> Color32 {
    let pr = pagerank.unwrap_or(0.0).clamp(0.0, 1.0);
    let uncovered = 1.0 - coverage_pct.unwrap_or(50.0).clamp(0.0, 100.0) / 100.0;
    let lint_factor = 1.0 + (clippy_count.unwrap_or(0) as f64 + 1.0).ln() / 5.0;

    // Risk peaks when: high centrality + uncovered + many lints
    let raw = pr * uncovered * lint_factor;
    // Normalize: clamp then map 0–2 to 0–1 (empirically, raw rarely exceeds 2)
    let t = (raw / 2.0).clamp(0.0, 1.0);

    // cool (low risk) → hot (high risk)
    let r = (30.0 + t * 225.0) as u8;
    let g = (180.0 * (1.0 - t)) as u8;
    let b = 40_u8;
    Color32::from_rgb(r, g, b)
}
```

### Pattern 7: Community Detection via Snapshot Import Graph

`pmat analyze graph-metrics` does NOT provide per-node community IDs in its JSON output — the `connected_components` field is a graph-level count. To implement "click a file to highlight its community," use the existing `Snapshot.import_graph` (already built during scan). The connected component containing a selected file can be found via BFS/DFS over the import edges already stored in `Arc<Snapshot>`:

```rust
// sentrux-core/src/app/state.rs — new field
pub community_highlight: Option<HashSet<String>>, // file paths in same component

// When user clicks a file in risk/rank mode:
fn compute_community(snapshot: &Snapshot, selected: &str) -> HashSet<String> {
    // BFS over snapshot import edges to find connected component
    let mut visited = HashSet::new();
    let mut queue = vec![selected.to_string()];
    while let Some(node) = queue.pop() {
        if visited.insert(node.clone()) {
            for edge in snapshot.import_edges.iter().filter(|e| e.from == node || e.to == node) {
                queue.push(if edge.from == node { edge.to.clone() } else { edge.from.clone() });
            }
        }
    }
    visited
}
```

This avoids any dependency on PMAT for community detection, works with the existing graph data, and is O(V+E) which is fast for typical project sizes.

### Anti-Patterns to Avoid

- **Joining graph-metrics nodes to snapshot by filename only:** Multiple files named `mod.rs` are common in Rust crates. If using graph-metrics data in the detail panel, use best-effort match (first match by basename) and note the limitation. Don't assert correctness.
- **Running cargo llvm-cov without --ignore-run-fail:** If any tests fail, llvm-cov exits without writing output. Always include this flag.
- **Running clippy without `--` separator:** Clippy lint flags go after `--` in the cargo invocation: `cargo clippy --message-format=json -- -W clippy::pedantic`. Without `--`, the flags are passed to cargo, not clippy.
- **Collecting all clippy diagnostic lines:** Many lines are `reason:compiler-artifact` or `reason:build-script-executed`. Only parse lines where `reason == "compiler-message"` and `message.level == "warning"`.
- **Adding Coverage/Risk ColorMode variants without serde handling:** The `#[serde(other)]` fallback is already on `Monochrome`. The new variants must appear BEFORE `Monochrome` in the enum definition to be serialized correctly (serde's `other` only fires for unknown variants, not for new ones).
- **Blocking the UI thread with llvm-cov:** Coverage can take 2 minutes. It must run on a background thread with progress reporting. The on-demand button triggers a background job, not a blocking subprocess.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| PageRank computation | Custom graph algorithm | `pmat analyze graph-metrics --format json` | PMAT's PageRank uses damping factor 0.85 (standard), handles convergence; reimplementing this correctly is non-trivial |
| Line coverage measurement | Manual test runner instrumentation | `cargo llvm-cov --json --summary-only` | LLVM coverage requires compiler instrumentation; there is no workaround |
| Clippy lint categorization | String-matching lint IDs manually | Fixed `lint_category()` mapping function | The 63 lint IDs found in sentrux are stable; a match table is correct, not fragile |
| Connected component detection | Custom community detection | BFS over `snapshot.import_graph` | The snapshot's import graph is already built during scan — no new computation needed |

pub struct CoverageSummaryMetric {
    pub count: u32,
    pub covered: u32,
    pub percent: f64,

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct FileCoverageSummary {
    pub lines: CoverageSummaryMetric,
    pub functions: CoverageSummaryMetric,

pub struct CoverageFileEntry {
    pub filename: String,             // absolute path
    pub summary: FileCoverageSummary,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct CoverageDataSection {
    pub files: Vec<CoverageFileEntry>,

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct CoverageOutput {
    pub data: Vec<CoverageDataSection>,

/// Aggregated report with relative-path lookup index.
#[derive(Debug, Clone)]
pub struct CoverageReport {
    pub files: Vec<CoverageFileEntry>,
    /// relative path (scan-root-relative) → index into files
    pub by_path: HashMap<String, usize>,
}

impl CoverageReport {
    pub fn from_output(output: CoverageOutput, scan_root: &str) -> Option<Self> {
        let files = output.data.into_iter().next()?.files;
        let mut by_path = HashMap::with_capacity(files.len());
        let root_with_sep = if scan_root.ends_with('/') {
            scan_root.to_string()

            format!("{}/", scan_root)
        };
        for (i, f) in files.iter().enumerate() {
            if let Some(rel) = f.filename.strip_prefix(&root_with_sep) {
                by_path.insert(rel.to_string(), i);
            }
        }
        Some(CoverageReport { files, by_path })
    }
}

        | "clippy::cast_possible_truncation"

## Common Pitfalls

### Pitfall 1: graph-metrics Returns Filenames, Not Full Paths

**What goes wrong:** `pmat analyze graph-metrics` node `name` is `"mod.rs"`, not `"sentrux-core/src/analysis/graph/mod.rs"`. A Rust workspace typically has dozens of `mod.rs` files.

**Why it happens:** PMAT's graph-metrics operates on a module-level graph where filenames are the node identifiers.

**How to avoid:** When looking up graph-metrics data for a selected file in the detail panel, extract the basename using `Path::file_name()` and use `by_filename` lookup. Accept that if multiple files share the same basename, the wrong one may be shown. Document this limitation in the panel with a note like "centrality shown for `mod.rs` (multiple matches possible)."

**Warning signs:** Detail panel shows PageRank 0.011 for every file (the uniform value PMAT returns when all nodes tie — common in disconnected graphs).

**Alternative if precision is needed:** Run `pmat analyze graph-metrics --path <specific_subdir>` per crate root to narrow the graph. This reduces filename collision. Trade-off: multiple subprocess calls.

### Pitfall 2: cargo-llvm-cov Fails Without --ignore-run-fail

**What goes wrong:** `cargo llvm-cov --json` exits with code 101 and writes NO output file when any test fails. sentrux currently has 27 pre-existing test failures (oracle failures from removed modules). Without `--ignore-run-fail`, coverage data is always unavailable.

**Why it happens:** cargo-llvm-cov's default behavior mirrors `cargo test` — fail on first test failure.

**How to avoid:** Always include `--ignore-run-fail`. The flag was confirmed to work with version 0.6.18.

**Warning signs:** `/tmp/sentrux_coverage_*.json` not created; `run_coverage` always returns `None`.

### Pitfall 3: Clippy Compile Cache

**What goes wrong:** `cargo clippy` is near-instant on a warm build (~5s) but takes the same time as a full build on a cold cache. If sentrux is first installed on a new machine, clippy during scan will block for 60+ seconds.

**Why it happens:** clippy compiles all dependencies before checking.

**How to avoid:** Run clippy on the scanner thread after the scan completes (same pattern as `run_pmat_tdg`). The first run is slow; subsequent runs are fast. Show a "Running clippy..." progress step.

**Warning signs:** First-ever scan taking 2+ minutes; user seeing "Running clippy..." for a long time.

### Pitfall 4: ColorMode Enum Serde — Adding Two New Variants

**What goes wrong:** Adding `Coverage` and `Risk` to the `ColorMode` enum without care breaks the `#[serde(other)]` placement rule. The `#[serde(other)]` attribute must be on the LAST variant. Adding new variants AFTER `Monochrome` would make them unreachable during deserialization.

**Why it happens:** `#[serde(other)]` consumes all unknown variants. Any variant defined after it is never matched.

**How to avoid:** Insert `Coverage` and `Risk` BEFORE `Monochrome` in the enum definition. `Monochrome` with `#[serde(other)]` must remain the final variant.

```rust
pub enum ColorMode {
    Language,
    Heat,
    Git,
    TdgGrade,
    Coverage,   // NEW — before Monochrome
    Risk,       // NEW — before Monochrome
    #[serde(other)]
    Monochrome, // MUST remain last
}
```

**Warning signs:** `Coverage` and `Risk` serialize/deserialize correctly in tests, but old prefs containing `"coverage"` or `"risk"` string cause panics in prod (not possible since `#[serde(other)]` captures everything).

### Pitfall 5: Coverage ColorMode Available Before Data Exists

**What goes wrong:** If `Coverage` is listed in `ColorMode::ALL`, the toolbar cycles to it and calls `file_color` with `ColorMode::Coverage`, but `state.coverage_report.is_none()`. The renderer falls back to monochrome silently — confusing to the user.

**Why it happens:** ColorMode is a simple enum cycle; the toolbar doesn't know which modes have data.

**How to avoid:** In the toolbar color mode button, disable the `Coverage` button (grey, no click) when `state.coverage_report.is_none()`. Show tooltip "Run coverage first (toolbar button)". The `Risk` mode can be available even without coverage (degrades to pagerank × clippy only).

**Warning signs:** Treemap appears monochrome after clicking Coverage with no explanation.

### Pitfall 6: Clippy JSON Contains Non-Warning Lines

**What goes wrong:** `cargo clippy --message-format=json` outputs hundreds of `compiler-artifact` and `build-script-executed` lines before the `compiler-message` lines. Attempting `serde_json::from_str` on the full output instead of line-by-line will fail.

**Why it happens:** cargo's `--message-format=json` produces NDJSON (newline-delimited JSON), not a JSON array.

**How to avoid:** Process byte stream line-by-line, skip lines where `obj["reason"] != "compiler-message"`. Only parse `message.level == "warning"` entries.

## Open Questions

1. **graph-metrics filename collision for multiple mod.rs files**
   - What we know: 88 nodes in sentrux, `by_filename` lookup works for uniquely-named files. Sentrux has multiple `mod.rs` files (in graph/, analysis/, etc.)
   - What's unclear: Whether the detail panel should show a disambiguation message or silently pick the first match
   - Recommendation: Pick first match (by PageRank descending as returned), add a note in the panel: "centrality data shown for highest-ranked file matching this name." Accept this limitation for v1.

2. **Clippy run timing: automatic during scan vs on-demand**
   - What we know: Clippy takes ~5s on a warm build in the sentrux workspace. Phase 2 already takes ~15s for PMAT. Adding 5s is a 30% increase.
   - Recommendation: Run automatically during scan (same as graph-metrics). The user decided "automatic if fast enough" and 5s on warm build qualifies. Show "Running clippy..." as a scan step.

3. **Risk ColorMode normalization constant**
   - What we know: The formula `raw = pr * (1 - cov%) * ln(lints+1)` needs a normalization divisor. For sentrux: max pr ≈ 0.011 (uniform), max raw ≈ 0.008 — too compressed.
   - What's unclear: Whether to normalize to project max (dynamic) or a fixed constant
   - Recommendation: Normalize to project max. Compute `max_risk = max(raw_risk_per_file)` then divide. This makes the "hottest" file always red, which matches user intent ("focus effort where it matters most").
